Pieces Auto TN

A quick friend-to-friend tour of what the project does and how to run it.

What it is

Pieces Auto TN is an auto-parts search system for Tunisia, running entirely through two Telegram bots. Customers find parts for their car; a mechanic/operator feeds the database from the official catalog. No website, no app — just chat.

Think of it as a chat-native version of a parts counter: the client types "Kia Picanto filtre a huile", the bot figures out which exact car they mean, looks up valid references in our database, and checks live prices/stock from the wholesaler.

The two bots

1. Client bot — for customers

  • Three ways to identify the car: photo of carte grise, VIN entry, or pick brand > model > year > engine with buttons.
  • Free text also works: "Peugeot 208 2019 courroie" or "quit de distribusion polo" (supports French, Arabic, and Tunisian franco-arabic spellings).
  • Returns the part references that fit, and — for the ones CDG carries — live price and stock.

2. Operator bot — for the mechanic / data-entry person

  • /ajouter_ref — send screenshots from PiecesAuto24 and the bot extracts vehicle + part + references automatically (LLM vision).
  • /vin — decode a VIN and map it to the right vehicle in DB.
  • /ref — consult the database (brand > model > year > engine > part, buttons only).
  • /dispo — live CDG availability + price check.
  • /stats, /guide — self-explanatory.

The ground rules (read these first)

  • PiecesAuto24 is the single source of truth for vehicle naming and part references.
  • CDG is the single source of truth for live price and stock (never cached — always scraped live).
  • The LLM never guesses part references. References only come from the database. The LLM only helps with text interpretation and OCR.
  • Zero free text for vehicle identification in the operator bot: always buttons sourced from the DB.

Database in one glance (6 tables)

  • vehicles — exact names from PiecesAuto24 (pa24_full_name).
  • vin_patterns — first 13 chars of a VIN → vehicle_id.
  • part_references — vehicle + part + brand + reference.
  • part_vehicle_compatibility — cross-reference compatible vehicles.
  • screenshots — local screenshot storage metadata.
  • requests_log — client request audit trail.

First-run setup

# 1. Install deps
uv sync

# 2. Copy .env.example → .env and fill in:
#    TELEGRAM_OPERATOR_BOT_TOKEN, TELEGRAM_CLIENT_BOT_TOKEN
#    OPENROUTER_API_KEY
#    CDG_URL, CDG_LOGIN, CDG_PASSWORD
#    DATABASE_URL   (Supabase Postgres)

# 3. Initialize the DB schema
uv run python3 -m src init-db

# 4. Seed from data/database.json (275+ vehicles, ~80k references)
uv run python3 -m src seed

# 5. Run both bots
uv run python3 -m src serve
serve auto-re-seeds on startup, so step 4 is optional after first run. Seed is idempotent — it only updates, never duplicates.

Important mental model

  • Display vs DB: buttons are cleaned up for humans, but callbacks carry an index into a session-cached raw list. The DB always sees the unmodified original string. See docs/ui_display_contract.md.
  • Sessions are in-memory: bot restart loses them. Acceptable for now.
  • Prices are never cached. Every /dispo is a live Playwright scrape. An asyncio lock serializes scraper calls across both bots.
  • Be simple, work incrementally. The codebase intentionally avoids over-engineering. See CLAUDE.md.

Things that will bite you (heads-up)

  • CDG session can expire — the scraper detects and re-logins.
  • CDG price extraction pairs prices to refs by index; layout shifts can misalign (known limitation).
  • First request after a long idle period can be slow if the asyncpg pool drops — the pool lazily re-opens.
  • database.json is maintained in a separate scraping session and re-seeded regularly. Don't edit it by hand.

If you remember only three things

  1. Two bots, one DB, one live scraper. That's the whole shape.
  2. LLM interprets words, not references. References are always DB-sourced.
  3. Run with uv run python3 -m src serve. Everything else follows.
